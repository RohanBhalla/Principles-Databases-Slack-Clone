# Design notes (Projects 1 + 2)

This document summarizes the schema from Project 1 and the small revisions made for Project 2, plus how the web app enforces access control and security.

## Core entities (Project 1)

- **Users**: register with email, username, nickname, password hash.
- **Workspaces**: created by a user; membership tracked in `workspace_members` with `is_admin`.
- **Channels**: belong to a workspace; types `public`, `private`, `direct`.\n+- **Membership**: `channel_members` controls visibility and posting rights.\n+- **Messages**: append-only posts in channels.\n+- **Invitations**: pending/accepted/rejected; targets *either* workspace or channel.\n+\n## Revisions in Project 2\n+\nTo support the UI and “cool extras”, the schema is extended with:\n+\n- `channel_reads(channel_id, user_id, last_read_at)`\n  - Tracks last time a user read a channel.\n  - Used to compute **unread badge counts**.\n-\n- `reactions(message_id, user_id, emoji)`\n  - Supports **emoji reactions** with counts.\n+\nAlso added:\n+\n- `users.username` is `UNIQUE`.\n- A partial unique index prevents duplicate **pending** invitations to the same target.\n\n## Access control (application-layer)\n\nThe app uses a single DB account (no per-user DB accounts). Access is enforced in two layers:\n\n- **Read queries** always join against membership tables (`workspace_members`, `channel_members`).\n- **Write operations** are done via stored procedures that validate membership and permissions before mutating tables.\n\nExamples:\n\n- Posting a message requires the user be a member of the workspace and channel (`post_message`).\n- Inviting to a workspace requires `is_admin = true` (`invite_to_workspace`).\n- Searching only returns messages visible to the user (`search_visible_messages`).\n\n## Transactions / concurrency\n\n- Each stored procedure runs atomically.\n- Invitation responses use `SELECT ... FOR UPDATE` to serialize handling of a pending invite.\n- Membership inserts use `ON CONFLICT DO NOTHING` so duplicate accept operations don’t create duplicate memberships.\n\n## Security\n\n- **SQL injection**: use prepared statements in the application; all write paths call stored procedures with bound parameters.\n- **XSS**: Jinja autoescape; user-provided strings are rendered escaped.\n- **CSRF**: session-scoped token required on all unsafe methods.\n- **Sessions**: signed session cookie via `SessionMiddleware`.\n*** End Patch"}]}"""
